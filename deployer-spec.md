# ТЗ: самописный сервис управления развёртыванием проектов

## Контекст

Есть личный VPS (2 GB RAM, 2 vCPU, 30 GB), на котором развёрнута инфраструктура через Docker Compose:

- **Reverse proxy:** Traefik (настроен, умеет labels, выдаёт сертификаты Let's Encrypt на поддомены)
- **SSO:** OAuth2-proxy перед сервисами
- **Мониторинг:** NetData
- **Логи:** Dozzle

На этой инфраструктуре крутятся пет-проекты на поддоменах. Сейчас деплой ручной: `git pull` + `docker compose up`. Нужен сервис, который это автоматизирует и даёт удобный web UI.

Готовые решения (Coolify, Dokploy) рассматривались и отброшены — слишком тяжёлые для имеющегося железа и тащат много ненужного функционала.

## Что нужно сделать ПЕРЕД написанием кода

**Critical:** перед началом реализации обязательно изучить существующие репозитории пользователя:

1. **Репозиторий с инфраструктурой** — посмотреть, как устроены Docker Compose файлы, какие сети используются, как подключены Traefik / OAuth2-proxy / NetData / Dozzle, какие конвенции для labels, какие переменные окружения используются. Это нужно, чтобы новый сервис встроился в существующий стек естественно, а не как чужеродное тело.

2. **Репозитории проектов** — посмотреть несколько реальных проектов, которые предполагается деплоить через этот сервис. Понять:
   - Как у них устроены compose-файлы (один сервис или несколько, какие зависимости)
   - Какие они используют labels для Traefik (или не используют)
   - Где у них env-переменные, что в них хранится
   - Какие из них приватные (нужны креды), какие публичные
   - Есть ли у них build-stage или используются готовые образы
   - Это подсветит граничные случаи: "проект с двумя поддоменами", "проект с volume на хост" и т.п.

3. **Стиль кода и UI** — взять стек и стилистику из существующего FastAPI-проекта пользователя. Не выдумывать новое: использовать те же подходы к структуре, логированию, обработке ошибок, что уже есть. UI (SPA) — взять стилистику из готового проекта, чтобы сервис визуально вписывался в личную экосистему.

После исследования — сначала предложить уточнённый план реализации с учётом найденных особенностей, и только потом писать код.

## Функциональные требования

### Источники проектов

Сервис должен уметь забирать проекты из трёх типов источников:

1. **Git-репозитории**
   - GitHub (публичные и приватные через personal access token)
   - Личный git-сервер (приватные через SSH-ключ)
   - Конкретный URL и тип авторизации задаются для каждого проекта индивидуально
   - Пользователь должен иметь возможность выбирать ветку **или** тег при деплое
   - Список доступных веток/тегов получать через `git ls-remote` без полного клонирования

2. **Docker Registry**
   - Публичные реестры (Docker Hub) и приватные с авторизацией
   - Пользователь выбирает версию (тег) образа при деплое
   - Список доступных тегов получать через Registry API v2 (`/v2/<name>/tags/list`)
   - В этом случае сам compose-файл хранится отдельно (либо введён через UI, либо подтянут из git)

3. **Локальная папка на сервере**
   - Просто путь к папке с готовым compose-файлом
   - Версионирования нет, деплой = `docker compose up` в этой папке
   - Полезно для одноразовых вещей и отладки

### Web UI

UI должен предоставлять следующие возможности:

1. **Просмотр всех запущенных контейнеров** — список со статусом, использованием ресурсов, ссылкой на логи (можно ссылкой на Dozzle, не дублировать функционал).

2. **Управление проектами:**
   - Список добавленных проектов
   - Добавление нового проекта (выбор типа источника, ввод параметров, сохранение)
   - Редактирование существующего (URL, креды, env-переменные, настройки поддомена)
   - Удаление проекта (с опцией "также остановить и удалить контейнеры")

3. **Управление версиями при деплое:**
   - Для git-источника — выпадающий список веток и тегов
   - Для registry-источника — выпадающий список тегов образа
   - Для локальной папки — версии нет, просто кнопка "deploy"
   - Кнопка "deploy" с выбранной версией; статус деплоя в реальном времени

4. **Управление поддоменами:**
   - Для каждого проекта можно задать: subdomain (например, `blog`), какой сервис из compose роутить (если их несколько), на какой внутренний порт
   - Сервис патчит compose через override-файл, добавляя нужные Traefik labels (см. ниже)

### Дополнительные функции (не упомянуты явно, но критичны)

5. **История деплоев и rollback:**
   - На каждый проект — лог последних N деплоев (версия, время, статус, логи процесса)
   - Возможность откатить на предыдущую успешную версию

6. **Управление переменными окружения:**
   - На каждый проект — редактируемый набор env-переменных
   - Они подкладываются в `.env` файл при деплое
   - Хранятся в БД сервиса, секретные значения — зашифрованно

7. **Webhook-эндпоинты для авто-деплоя:**
   - Каждый проект может получить уникальный webhook URL
   - При POST на него с правильным секретом (HMAC от GitHub) запускается деплой указанной ветки
   - Это позволит сделать push-to-deploy

8. **Логи процесса деплоя:**
   - Сохранять stdout/stderr операций `git pull`, `docker compose up` и т.п.
   - Показывать в UI в реалтайме (через Server-Sent Events или WebSocket)
   - Отделять от логов самого приложения (для них есть Dozzle)

9. **Health check после деплоя:**
   - После `docker compose up -d` — проверка, что контейнер действительно работает (например, не упал через 10 секунд)
   - Если упал — пометить деплой как failed, не trigger-ить уведомление о success

10. **Управление секретами:**
    - SSH-ключи для git, токены для GitHub, креды для registry
    - Хранятся зашифрованно в БД
    - В UI можно добавить/удалить, но просмотр значения после сохранения не должен быть доступен

11. **Параллельность:**
    - Деплои одного проекта должны идти последовательно (lock на проект)
    - Деплои разных проектов могут идти параллельно
    - Использовать очередь задач (даже простую in-memory)

## Технические требования и ограничения

### Стек

- **Backend:** Python + FastAPI (как в остальных проектах пользователя)
- **БД:** SQLite (одного файла достаточно для одного юзера и десятка проектов; никакого Postgres)
- **Очередь:** в памяти (`asyncio.Queue`), без Redis
- **Frontend:** SPA в стилистике существующих проектов пользователя (изучить и переиспользовать стиль)
- **Docker SDK:** `docker` (docker-py) для Python
- **Git:** субпроцесс `git` или библиотека (например, `pygit2` / `GitPython`)

### Ресурсы и упаковка

- Сервис должен потреблять минимум ресурсов: idle < 100 MB RAM
- Запускается в Docker Compose как часть существующей инфраструктуры
- Использует `tecnativa/docker-socket-proxy` для управления Docker (НЕ монтировать docker.sock напрямую)
- За OAuth2-proxy для аутентификации — никакой своей системы пользователей не делать
- Слушает только на внутренней сети, наружу — через существующий Traefik

### Работа с Docker

- Управление через Docker socket proxy с минимально необходимыми разрешениями
- Каждый проект — отдельный compose project name (`-p <project_name>`)
- Команды через `docker compose -f base.yml -f override.yml -p <name> up -d`
- Чистое разделение: исходный compose-файл проекта не модифицируется, всё кастомное идёт в override

### Управление поддоменами через override-файлы

Подход: брать оригинальный `docker-compose.yml` проекта без изменений, генерировать рядом `docker-compose.override.yml` с нужными Traefik-метками и подключением к сети Traefik. При деплое использовать оба файла.

Шаблон override-файла (адаптировать под существующие конвенции из инфраструктурного репо):

```yaml
services:
  {service_name}:
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.{project}.rule=Host(`{subdomain}.example.com`)"
      - "traefik.http.routers.{project}.tls.certresolver=letsencrypt"
      - "traefik.http.services.{project}.loadbalancer.server.port={port}"
    networks:
      - traefik
networks:
  traefik:
    external: true
```

Точные имена сетей, certresolver, доменов — взять из существующей инфраструктуры (это причина, почему важно сначала её изучить).

### Структура файлов на хосте

```
/var/lib/deployer/  (или другой путь, согласовать)
  projects/
    <project_name>/
      source/                   # git clone или скачанный compose
      docker-compose.override.yml  # генерируется сервисом
      .env                      # генерируется из БД
      deploy.log
  state.db                       # SQLite
  secrets/                       # зашифрованные секреты, chmod 600
```

### Безопасность

- Сервис за OAuth2-proxy (никакой своей аутентификации)
- Webhook endpoints — отдельно от UI, защищены HMAC-подписью
- Docker API — через socket-proxy с whitelist разрешений
- Секреты в БД зашифрованы (ключ — из переменной окружения)
- Никаких "выполнить произвольную команду из UI" — только заранее заданные операции

## Схема БД (предварительная)

```
projects
  id, name, source_type, source_url, source_credentials_id (FK secrets)
  current_version, current_compose_path
  service_name, subdomain, internal_port
  env_vars (JSON, шифрованный)
  created_at, updated_at

deployments
  id, project_id (FK), version, status (pending/running/success/failed)
  started_at, finished_at, log
  triggered_by (manual/webhook)

secrets
  id, name, type (ssh_key/git_token/registry_creds)
  encrypted_value
  created_at

webhooks
  id, project_id (FK), token, secret, branch_filter
```

## Этапы реализации

Двигаться итерациями, на каждой получать работающий результат:

**Этап 0:** изучить существующие репозитории (инфра + проекты) и существующий FastAPI-стек пользователя. Уточнить план с учётом найденного.

**Этап 1:** скелет — FastAPI + SQLite + минимальный UI. Можно создать проект из локальной папки, кнопка "deploy" делает `docker compose up`. Без секретов, без override-файлов, всё в одном бинарнике на хосте (для отладки).

**Этап 2:** добавить GitSource и RegistrySource, шифрование секретов, выбор версии при деплое.

**Этап 3:** override-файлы, управление поддоменами, интеграция с Traefik из существующей инфраструктуры.

**Этап 4:** упаковка в Docker, socket-proxy, размещение за OAuth2-proxy. Webhook endpoints для авто-деплоя.

**Этап 5:** операционные функции — история деплоев, rollback, health checks, реалтайм-логи деплоя.

## Что НЕ нужно делать

- Не делать свою систему пользователей и ролей (есть SSO через OAuth2-proxy)
- Не делать свой просмотр логов приложений (есть Dozzle)
- Не делать свои метрики (есть NetData)
- Не дублировать reverse proxy / SSL — Traefik уже всё делает
- Не делать поддержку Kubernetes, Docker Swarm, multi-server — только локальный Docker
- Не делать build-серверы, превью-окружения, branch deploys — только указанный пользователем deploy
- Не тащить лишних зависимостей: Postgres, Redis, очереди — всё это не нужно для одного юзера

## Открытые вопросы (уточнить у пользователя ПОСЛЕ изучения репозиториев)

1. Точное имя домена и subdomain-конвенция (например, `*.example.com` или другая схема)
2. Имя сети Traefik в существующей инфраструктуре
3. Имя certresolver Traefik
4. Где должен жить `/var/lib/deployer/` — на хосте или в named volume
5. Существующий стиль FastAPI-проектов: какая структура, какой ORM (или sqlite напрямую), какой паттерн для роутов
6. Стиль существующего SPA: какой фреймворк (React/Vue/Svelte/что-то ещё), какая UI-библиотека, общие компоненты
7. Как сейчас устроены секреты в инфраструктуре — есть ли уже какой-то общий подход (например, .env файлы, Docker secrets), стоит ли его переиспользовать
